Grey Box Testing

Hybrid testing approach with partial knowledge of internal structure.

What is Grey Box Testing?

Grey Box Testing (also known as Gray Box Testing) is a software testing technique that combines elements of both White Box and Black Box testing. The tester has partial knowledge of the internal workings of the application, such as access to design documents, database schemas, or limited source code, while testing from an external user perspective.

Key Characteristics

Aspect Description
Knowledge Required Partial access to internal structure, design docs, database schema
Testing Focus Both external behavior and internal data flow/logic
Tester Profile Testers with some development/architectural knowledge
Approach Combines user perspective with internal understanding

Types of Grey Box Testing

1 Matrix Testing

Examining status variables and runtime variables during program execution to identify unused or unoptimized variables.

2 Pattern Testing

Analyzing historical data to identify patterns of defects and focusing testing on similar areas.

3 Orthogonal Array Testing

Statistical testing approach for complex applications with multiple inputs and configurations.

4 Regression Testing

Testing after code changes to ensure modifications haven't introduced new defects.

Database Testing in Grey Box

Grey Box Testing is particularly effective for database testing where testers have knowledge of the database schema but test from the application interface:

  • Semantic Testing: Validate database mappings and front-end/back-end consistency
  • Atomicity: Verify transaction completeness (all or nothing)
  • Consistency: Ensure database remains valid after transactions
  • Isolation: Test concurrent transaction handling
  • Durability: Verify committed data survives system failures
  • Constraint Checking: Validate primary keys, foreign keys, and constraints
  • Data Integrity: Verify data accuracy and consistency

Grey Box Testing Commands

API Testing

# Test with authentication
curl -X GET https://api.target.com/v1/users \
  -H "Authorization: Bearer TOKEN" \
  -H "Content-Type: application/json"

# Test role-based access
curl -X GET https://api.target.com/v1/admin/users \
  -H "Authorization: Bearer USER_TOKEN"

# Test data validation
curl -X POST https://api.target.com/v1/users \
  -H "Content-Type: application/json" \
  -d '{"email":"test@test.com","role":"admin"}'

Database Testing

# SQL Injection with partial knowledge
' UNION SELECT username, password FROM users--
' OR '1'='1' --
' AND 1=2 UNION SELECT null, version()--

# Time-based blind SQLi
' AND (SELECT * FROM (SELECT(SLEEP(5)))a)--

# Error-based SQLi
' AND extractvalue(1, concat(0x7e, (SELECT @@version)))--

Web Testing

# Test hidden fields
ffuf -u target.com/page -d "FUZZ=test" -w hidden-params.txt

# Test with session cookies
curl -X POST target.com/action \
  -b "sessionid=VALUE; csrf_token=VALUE" \
  -d "action=delete&id=123"

# Bypass client-side validation
curl -X POST target.com/api/submit \
  -H "Content-Type: application/json" \
  -d '{"amount":-100,"price":0}'

Grey Box Testing Checklist

  • Review system architecture and design documents
  • Analyze database schema and relationships
  • Test API endpoints with different authentication levels
  • Verify role-based access control (RBAC)
  • Test data flow between frontend and backend
  • Validate ORM queries and SQL generation
  • Test transaction handling and rollback mechanisms
  • Verify caching mechanisms and invalidation
  • Test session management and timeout handling
  • Validate input validation at all layers
  • Test business logic vulnerabilities
  • Verify audit logging functionality
  • Test integration points between modules
  • Validate configuration management

Grey Box vs White Box vs Black Box

Aspect Black Box Grey Box White Box
Internal Knowledge None Partial Full
Test Basis Requirements Requirements + Design Code + Design
Testers QA/Security Testers Testers + Developers Developers
Time Required High Medium Low-Medium
Code Coverage Unknown Partial Measurable
Real-world Simulation Highest High Low

Common Vulnerabilities Found

Vulnerability Discovery Method
IDOR (Insecure Direct Object Reference) Modify IDs after analyzing URL patterns
Business Logic Flaws Understand workflow then test edge cases
Privilege Escalation Test with different role tokens
Mass Assignment Add extra fields based on schema knowledge
Timing Attacks Analyze response times for different inputs
Race Conditions Send parallel requests to critical endpoints

Best Practices

  • Use design documents to identify critical test areas
  • Leverage database schema for injection testing
  • Test with multiple user roles and permissions
  • Focus on integration points between components
  • Test data consistency across frontend and backend
  • Validate all API endpoints with proper authentication
  • Use knowledge of tech stack to target specific vulnerabilities
  • Combine automated scanning with manual testing
