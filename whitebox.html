White Box Testing

Testing with full knowledge of internal structure, design, and implementation.

What is White Box Testing?

White Box Testing (also known as Clear Box Testing, Glass Box Testing, or Structural Testing) is a testing technique that examines the internal structure, design, and coding of software to verify input-output flow and improve design, usability, and security. The tester has complete visibility into the application's internal workings.

Key Characteristics

Aspect Description
Knowledge Required Full access to source code, architecture, and design documents
Testing Focus Internal logic, code structure, data flow, and control flow
Tester Profile Developers, security analysts with programming knowledge
Coverage Code coverage, path coverage, branch coverage

Types of White Box Testing

1 Unit Testing

Testing individual units or components of source code. Each unit is tested separately to ensure it works correctly.

2 Integration Testing

Testing the interaction between integrated units/modules to detect interface defects.

3 Penetration Testing (Code Review)

Manual or automated review of source code to find security vulnerabilities.

4 Mutation Testing

Modifying code to check test suite robustness and effectiveness.

Code Coverage Techniques

  • Statement Coverage: Every line of code is executed at least once
  • Branch Coverage: Every decision branch (true/false) is executed
  • Path Coverage: All possible paths through the code are tested
  • Condition Coverage: Each boolean condition is evaluated as true and false
  • Function Coverage: Every function is called and executed
  • Loop Coverage: All loops are executed 0, 1, and multiple times

White Box Testing Commands

Static Analysis

# SonarQube Scanner
sonar-scanner -Dsonar.projectKey=myproject -Dsonar.sources=.

# Bandit (Python Security)
bandit -r /path/to/python/code

# Brakeman (Rails Security)
brakeman -A -w1 --format text

# Semgrep (Multi-language)
semgrep --config=auto .

# CodeQL (GitHub Security)
codeql database create --language=javascript /db
codeql analyze /db --format=sarifv2.1.0 --output=results.sarif

Code Coverage

# JaCoCo (Java)
java -javaagent:jacocoagent.jar -jar myapp.jar

# Coverage.py (Python)
coverage run -m pytest
coverage report
coverage html

# NYC (Node.js)
nyc npm test
nyc report --reporter=html

Unit Testing

# JUnit (Java)
mvn test

# Pytest (Python)
pytest -v --cov=myapp tests/

# Jest (JavaScript)
npm test -- --coverage

# Go Testing
go test -v -cover ./...

White Box Testing Checklist

  • Review all source code for security vulnerabilities
  • Check for hardcoded credentials and secrets
  • Verify input validation and sanitization
  • Test all conditional branches and paths
  • Review authentication and authorization logic
  • Check for SQL injection vulnerabilities
  • Verify secure cryptographic implementations
  • Test error handling and exception management
  • Review third-party dependencies
  • Ensure secure session management
  • Validate file upload functionality
  • Check for insecure deserialization
  • Review logging and audit mechanisms
  • Test API endpoints thoroughly

Common Vulnerabilities Found

Vulnerability Description Example
Hardcoded Credentials Passwords/API keys in source code password = "admin123"
SQL Injection Unsanitized user input in queries "SELECT * FROM users WHERE id = " + id
XSS Unescaped output in templates <div>{{ user_input }}</div>
Insecure Crypto Weak encryption algorithms MD5, SHA1 for passwords
Path Traversal Unsanitized file paths open(request.GET['file'])
Command Injection Unsanitized system commands os.system("ping " + host)

Best Practices

  • Use static analysis tools in CI/CD pipeline
  • Achieve at least 80% code coverage
  • Regular security code reviews
  • Use parameterized queries to prevent SQL injection
  • Implement proper input validation at all entry points
  • Use secure coding guidelines (OWASP, CERT)
  • Regular dependency scanning and updates
  • Implement proper error handling without information disclosure
